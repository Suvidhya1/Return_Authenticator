# ReturnAuthenticator — Return & Refund Predictor

A full-stack machine learning application that classifies incoming product-return
requests as **genuine**, **avoidable**, or **fraudulent**, and recommends an action
(instant refund, wait for the item, or route to a human agent) before the return is
processed.

The focus of this project is the **machine learning pipeline** — feature engineering,
model comparison, handling class imbalance, and honest evaluation — wrapped in a small
web app so the model can be used interactively.

---

## How it works

```
React frontend  ──►  Node/Express API  ──►  Python ML service (Flask)
   (portal)            (persistence,            (loads trained model,
                        request routing)         returns risk score)
                            │
                            ▼
                         MongoDB
                   (stored request history)
```

The web API and the ML service are **separate processes on purpose**. In real systems
the model is usually served independently so it can be retrained, scaled, or swapped
without redeploying the whole application.

## The machine learning

**Problem type:** multi-class supervised classification (3 classes).

**Features used:**
- `order_value` — total value of the order
- `product_category` — electronics, clothing, etc. (one-hot encoded)
- `customer_return_rate` — share of this customer's past orders that were returned
- `days_since_purchase` — timing of the return
- `prior_orders` — how established the customer is
- `return_reason` — stated reason (one-hot encoded)
- `is_high_value` — flag for unusually expensive items

**Pipeline:**
1. **Preprocessing** — numeric features are standardized; categorical features are
   one-hot encoded. Both steps live inside a scikit-learn `Pipeline` so the exact same
   transformation is applied at training and prediction time (no train/serve skew).
2. **Models compared** — a **Logistic Regression** baseline vs. a **Random Forest**.
   The Random Forest wins because the fraud signal is non-linear and involves feature
   interactions (e.g. high value *and* high return rate *and* unusual timing).
3. **Class imbalance** — fraud is only ~10% of cases, so both models use
   `class_weight="balanced"`, and the models are selected on **macro-F1** rather than
   plain accuracy (a model that predicts "genuine" for everything would score ~62%
   accuracy but be useless).
4. **Evaluation** — per-class precision/recall, confusion matrix, and feature
   importances to confirm the model learned sensible patterns.

**Why these choices matter:** the most important features turn out to be
`customer_return_rate`, `days_since_purchase`, and `prior_orders` — which is exactly
what a fraud analyst would look at. That sanity check is part of the point.

> **Note on the dataset:** the data is *synthetic*, generated by `generate_data.py` to
> mimic the structure of real return-fraud data (rare fraud, learnable-but-noisy signal).
> Reported metrics reflect synthetic data and would differ on real-world data.

## Running it locally

You need **Python 3.9+** and **Node.js 18+**. MongoDB is optional.

### 1. ML service
```bash
cd ml
pip install -r requirements.txt
python generate_data.py     # creates data/returns.csv
python train.py             # trains, evaluates, saves model.joblib
python predict_service.py   # serves on http://localhost:5001
```

### 2. Backend (new terminal)
```bash
cd server
npm install
cp .env.example .env        # optionally add a MONGODB_URI
npm start                   # serves on http://localhost:5000
```

### 3. Frontend (new terminal)
```bash
cd client
npm install
npm run dev                 # opens http://localhost:5173
```

Open the frontend, fill in a return request, and submit. Try a high-value item with a
high return rate and a 1-day return window to see it get flagged.
